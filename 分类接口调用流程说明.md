## 实际调用流程

### 1. 首次打开分类选择弹窗
```javascript
// 只请求一级分类
fetchLevel1Categories() 
  ↓
categoryApi.getCategoryList({
  pageNum: 1,
  pageSize: 100,
  parentId: 0  // ✅ 只获取一级分类
})
```

**结果：** 只加载一级分类数据，不加载任何二级分类

### 2. 用户点击某个一级分类
```javascript
selectLevel1(cat)  // 用户点击了一级分类
  ↓
fetchLevel2Categories(cat.id)  // ✅ 按需加载该分类下的二级分类
  ↓
categoryApi.getCategoryList({
  pageNum: 1,
  pageSize: 100,
  parentId: cat.id  // ✅ 只获取该一级分类下的二级分类
})
```

**结果：** 只加载被点击的一级分类下的二级分类

### 3. 缓存机制
```javascript
// 如果已经加载过，直接使用缓存
if (level2CategoriesMap[parentId]) {
  console.log(`✅ 二级分类已缓存，直接使用: parentId=${parentId}`);
  return;
}
```

**结果：** 同一个一级分类的二级分类只会请求一次

## 为什么会显示很多一级分类？

如果你看到很多一级分类，可能的原因：

### 原因 1：后端返回的数据本身就很多
- 后端数据库中确实有很多 `parentId=0` 的分类
- 这不是前端的问题，是后端数据的问题

### 原因 2：后端没有正确过滤 parentId
- 前端请求：`parentId=0`
- 后端返回：包含了 `parentId` 不为 0 的数据（二级分类也返回了）
- 这是后端接口的 bug

### 原因 3：数据重复
- 后端数据库中有重复的分类数据
- 需要后端清理数据

### 检查要点：

1. **查看 ParentId 字段**
   - 一级分类：`ParentId: 0` 或 `ParentId: null`
   - 二级分类：`ParentId: 1, 2, 3...`（一级分类的ID）

2. **如果看到 ParentId 不为 0 的数据**
   - 说明后端接口没有正确过滤
   - 需要后端修复接口

3. **如果所有数据的 ParentId 都是 0**
   - 说明后端数据库中确实有这么多一级分类
   - 需要后端清理数据

## 总结

**前端代码：** ✅ 完全正确
- 首次只请求一级分类（`parentId=0`）
- 点击后才请求二级分类（`parentId=一级分类ID`）
- 有缓存机制，避免重复请求

**问题可能在：**
- ❌ 后端接口没有正确过滤 `parentId`
- ❌ 后端数据库中有太多一级分类
- ❌ 后端数据库中有重复数据
