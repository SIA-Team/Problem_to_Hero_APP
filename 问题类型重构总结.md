# 问题类型重构总结

## 问题描述
之前的代码使用字符串来表示问题类型，然后在发送API请求时进行映射转换，这增加了不必要的复杂性和出错的可能性。

## 重构前的设计
```javascript
// UI层使用字符串
const questionTypes = [
  { id: 'free', name: '公开问题' },
  { id: 'reward', name: '悬赏问题' },
  { id: 'targeted', name: '定向问题' },
];

// 状态使用字符串
const [selectedType, setSelectedType] = useState('free');

// 发送请求时需要映射
type: selectedType === 'free' ? 0 : selectedType === 'reward' ? 1 : 2
```

## 重构后的设计
```javascript
// UI层直接使用数字
const questionTypes = [
  { id: 0, name: '公开问题' },    // 0 = 公开问题
  { id: 1, name: '悬赏问题' },    // 1 = 悬赏问题
  { id: 2, name: '定向问题' },    // 2 = 定向问题
];

// 状态直接使用数字
const [selectedType, setSelectedType] = useState(0);

// 发送请求时直接使用，无需映射
type: selectedType
```

## 重构的好处

### 1. 简化代码
- **移除了映射逻辑**：不再需要 `selectedType === 'free' ? 0 : ...` 这样的三元运算符
- **减少了代码量**：每个使用类型的地方都简化了
- **提高了可读性**：直接使用数字，与API保持一致

### 2. 减少错误
- **消除了映射错误**：不会因为映射逻辑写错而导致类型错误
- **类型一致性**：UI层和API层使用相同的数据类型
- **减少了调试复杂度**：不需要跟踪映射过程

### 3. 性能提升
- **减少了运行时计算**：不需要在每次请求时进行映射计算
- **内存效率**：数字比字符串占用更少内存

## 修改的文件和位置

### src/screens/PublishScreen.js
1. **questionTypes 数组**：id 从字符串改为数字
2. **selectedType 状态**：初始值从 'free' 改为 0
3. **所有条件判断**：从字符串比较改为数字比较
4. **UI 条件渲染**：更新所有 selectedType 的条件判断
5. **API 请求构建**：移除映射逻辑，直接使用 selectedType

## 类型对应关系
```
0 = 公开问题 (free)
1 = 悬赏问题 (reward)  
2 = 定向问题 (targeted)
```

## 测试建议
1. **功能测试**：确保三种问题类型都能正常选择和发布
2. **UI测试**：确保问题类型选择器显示正确
3. **API测试**：确认发送给服务器的 type 字段值正确
4. **边界测试**：测试类型切换时的表单状态变化

## 向后兼容性
这次重构是内部实现的改变，不影响：
- API接口规范
- 用户界面显示
- 数据库存储格式

## 结论
通过这次重构，我们：
- **消除了不必要的抽象层**
- **简化了代码逻辑**
- **提高了代码的可维护性**
- **减少了出错的可能性**

这是一个很好的例子，说明了"保持简单"(KISS原则)的重要性。当UI层和API层使用相同的数据格式时，就不需要额外的映射层。
