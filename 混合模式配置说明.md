# 混合模式配置说明

## 功能说明

支持部分接口使用真实服务器，部分接口使用 Mock 服务器的混合模式。

## 使用场景

### 场景 1：部分接口已完成

```
✅ 认证接口（登录、注册）- 后端已完成 → 使用真实服务器
❌ 用户资料接口 - 后端开发中 → 使用 Mock
❌ 问题列表接口 - 后端开发中 → 使用 Mock
```

### 场景 2：逐步迁移

```
第一周：所有接口使用 Mock
第二周：认证接口切换到真实服务器
第三周：用户接口切换到真实服务器
第四周：所有接口切换到真实服务器
```

### 场景 3：测试特定功能

```
✅ 大部分接口使用真实服务器
❌ 某个有问题的接口临时切换到 Mock 进行调试
```

## 配置方式

### 1. 打开配置文件

编辑 `src/config/env.js`

### 2. 配置接口模式

```javascript
const API_MODE_CONFIG = {
  // 认证相关接口
  '/app/user/auth/register': 'real',      // 使用真实服务器
  '/app/user/auth/login': 'real',         // 使用真实服务器
  '/app/user/auth/token-login': 'real',   // 使用真实服务器
  
  // 用户相关接口
  '/app/user/profile': 'mock',            // 使用 Mock
  '/app/user/profile/me': 'mock',         // 使用 Mock
  
  // 问题相关接口
  '/questions': 'mock',                   // 使用 Mock
  
  // 其他接口不配置，跟随全局 USE_MOCK 设置
};
```

### 3. 配置选项

| 值 | 说明 | 效果 |
|---|------|------|
| `'real'` | 使用真实服务器 | 请求发送到 `http://123.144.100.10:30560/qa-hero-app-user` |
| `'mock'` | 使用 Mock 服务器 | 请求发送到 `https://m1.apifoxmock.com/...` |
| 不配置 | 跟随全局设置 | 根据 `USE_MOCK` 决定 |

## 工作原理

### 请求拦截器

在 `apiClient.js` 的请求拦截器中：

```javascript
// 判断当前接口是否使用 Mock
const useMock = shouldUseMock(config.url);

// 动态设置 baseURL
if (useMock) {
  config.baseURL = MOCK_BASE_URL;  // Mock 服务器
} else {
  config.baseURL = REAL_BASE_URL;  // 真实服务器
}
```

### 判断逻辑

```javascript
export const shouldUseMock = (url) => {
  // 如果有具体配置，使用具体配置
  if (API_MODE_CONFIG[url] !== undefined) {
    return API_MODE_CONFIG[url] === 'mock';
  }
  
  // 否则跟随全局配置
  return USE_MOCK;
};
```

## 配置示例

### 示例 1：所有接口使用 Mock

```javascript
const USE_MOCK = true;

const API_MODE_CONFIG = {
  // 不配置任何接口
};
```

**效果**：所有接口都使用 Mock 服务器

### 示例 2：所有接口使用真实服务器

```javascript
const USE_MOCK = false;

const API_MODE_CONFIG = {
  // 不配置任何接口
};
```

**效果**：所有接口都使用真实服务器

### 示例 3：认证接口使用真实服务器，其他使用 Mock

```javascript
const USE_MOCK = true;  // 全局使用 Mock

const API_MODE_CONFIG = {
  // 认证接口使用真实服务器
  '/app/user/auth/register': 'real',
  '/app/user/auth/login': 'real',
  '/app/user/auth/token-login': 'real',
  
  // 其他接口不配置，跟随全局 USE_MOCK = true，使用 Mock
};
```

**效果**：
- 认证接口 → 真实服务器
- 其他接口 → Mock 服务器

### 示例 4：大部分使用真实服务器，个别使用 Mock

```javascript
const USE_MOCK = false;  // 全局使用真实服务器

const API_MODE_CONFIG = {
  // 个别接口使用 Mock
  '/questions': 'mock',
  '/app/user/profile': 'mock',
  
  // 其他接口不配置，跟随全局 USE_MOCK = false，使用真实服务器
};
```

**效果**：
- 问题接口、用户资料接口 → Mock 服务器
- 其他接口 → 真实服务器

## 注意事项

### 1. 接口路径必须完全匹配

```javascript
// ✅ 正确
'/app/user/auth/login': 'real',

// ❌ 错误（路径不完整）
'/login': 'real',

// ❌ 错误（多了参数）
'/app/user/auth/login?username=test': 'real',
```

### 2. 动态路径需要特殊处理

对于带参数的路径（如 `/app/user/profile/123`），需要在配置中使用基础路径：

```javascript
// 配置基础路径
'/app/user/profile': 'mock',

// 实际请求 /app/user/profile/123 时
// 需要在 shouldUseMock 中处理
```

### 3. 真实服务器必须可访问

配置为 `'real'` 的接口，真实服务器必须：
- ✅ 已部署
- ✅ 网络可访问
- ✅ 接口已实现

否则会请求失败。

### 4. Mock 数据一致性

使用 Mock 的接口仍然会使用 `LocalMockService` 保持数据一致性。

## 实际应用流程

### 第一阶段：全部 Mock

```javascript
const USE_MOCK = true;
const API_MODE_CONFIG = {};
```

- 前端独立开发
- 所有接口使用 Mock

### 第二阶段：认证接口上线

```javascript
const USE_MOCK = true;
const API_MODE_CONFIG = {
  '/app/user/auth/register': 'real',
  '/app/user/auth/login': 'real',
  '/app/user/auth/token-login': 'real',
};
```

- 认证接口切换到真实服务器
- 其他接口继续使用 Mock

### 第三阶段：用户接口上线

```javascript
const USE_MOCK = true;
const API_MODE_CONFIG = {
  '/app/user/auth/register': 'real',
  '/app/user/auth/login': 'real',
  '/app/user/auth/token-login': 'real',
  '/app/user/profile': 'real',
  '/app/user/profile/me': 'real',
};
```

- 认证 + 用户接口使用真实服务器
- 其他接口继续使用 Mock

### 第四阶段：全部上线

```javascript
const USE_MOCK = false;
const API_MODE_CONFIG = {};
```

- 所有接口使用真实服务器
- 删除或注释 API_MODE_CONFIG

## 优势

✅ **灵活切换**：可以单独控制每个接口  
✅ **逐步迁移**：不需要一次性切换所有接口  
✅ **易于调试**：可以临时切换某个接口到 Mock  
✅ **团队协作**：前后端可以独立开发  
✅ **降低风险**：逐步验证，减少一次性切换的风险

## 总结

混合模式配置让你可以：
- 部分接口使用真实服务器
- 部分接口使用 Mock 服务器
- 灵活控制每个接口的数据来源
- 逐步从 Mock 迁移到真实服务器
