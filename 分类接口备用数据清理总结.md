# 分类接口备用数据清理总结

## 问题描述
用户发现分类接口 `/app/content/category/list` 返回的一级分类数量不正确，显示了很多个分类，怀疑有备用数据混入。

## 问题原因
在 `PublishScreen.js` 中存在硬编码的备用数据 `categoryData`，包含：
- 3个一级分类（国家、行业、个人）
- 多个二级分类

这些备用数据在以下场景会被使用：
1. 一级分类加载失败时
2. 二级分类加载失败时
3. 渲染二级分类列表时

## 修复内容

### 1. 删除了硬编码的 categoryData 常量
```javascript
// 已删除
const categoryData = {
  level1: [...],
  level2: {...}
};
```

### 2. 修改了一级分类加载失败的处理逻辑
**修改前：**
```javascript
setLevel1Categories(categories);
```
失败时会使用 `categoryData.level1` 作为备用数据

**修改后：**
```javascript
setLevel1Categories([]); // 失败时保持空数组
```

### 3. 修改了二级分类加载失败的处理逻辑
**修改前：**
```javascript
setLevel2CategoriesMap(prev => ({
  ...prev,
  [parentId]: categoryData.level2[parentId] || [],
}));
```

**修改后：**
```javascript
setLevel2CategoriesMap(prev => ({
  ...prev,
  [parentId]: [], // 失败时保持空数组
}));
```

### 4. 修改了 getDisplayCategoryData 函数
**修改前：**
```javascript
const getDisplayCategoryData = () => {
  return {
    level1: level1Categories.length > 0 ? level1Categories : categoryData.level1,
    level2: level2CategoriesMap,
  };
};
```

**修改后：**
```javascript
const getDisplayCategoryData = () => {
  return {
    level1: level1Categories, // 只使用接口数据
    level2: level2CategoriesMap,
  };
};
```

### 5. 删除了 transformLocalCategoryData 函数
这个函数用于转换本地备用数据，现在已经不需要了。

### 6. 修改了渲染二级分类的逻辑
**修改前：**
```javascript
{(level2CategoriesMap[tempSelectedLevel1?.id] || categoryData.level2[tempSelectedLevel1?.id] || []).map(cat => (
  // ...
))}
```

**修改后：**
```javascript
{(level2CategoriesMap[tempSelectedLevel1?.id] || []).map(cat => (
  // ...
))}
```

## 验证结果
- ✅ 所有 `categoryData` 备用数据已完全移除
- ✅ 一级分类只显示接口返回的数据
- ✅ 二级分类只显示接口返回的数据
- ✅ 加载失败时显示空列表，不会混入备用数据

## 影响范围
- **文件：** `src/screens/PublishScreen.js`
- **功能：** 发布问题页面的分类选择功能
- **用户体验：** 
  - 分类数据完全来自后端接口
  - 如果接口失败，会显示空列表而不是备用数据
  - 便于调试和排查接口问题

## 注意事项
1. 所有真实接口都不再使用备用数据
2. 如果接口返回失败，用户将看到空的分类列表
3. 建议后端确保接口稳定性和数据准确性
4. 前端已添加详细的日志输出，便于调试

## 相关文件

- `src/services/api/categoryApi.js` - 分类接口

- `src/utils/dataLoader.js` - 数据加载器（已过滤 status=0 的数据）
