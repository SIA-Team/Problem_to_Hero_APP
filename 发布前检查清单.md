# 发布前检查清单

## 目标

这份清单用于减少不必要的热更新消耗，并尽量保证：

- 模拟器测试结果和热更新后的结果保持一致
- 真正需要热更新时再发布
- 原生改动和纯 JS 改动区分处理

---

## 一、日常开发调试

先用开发模式快速改功能：

```powershell
npm run start:dev
```

如果局域网连接不稳定，可以改用：

```powershell
npm run start:dev:tunnel
```

这一阶段的目标是：

- 快速看到修改结果
- 快速验证主流程
- 不做热更新

---

## 二、发布前最终验证

在准备发布热更新前，一定要再用一次接近生产环境的模式验证：

```powershell
npx expo start --dev-client --no-dev --minify -c
```

说明：

- `--no-dev`：关闭开发态逻辑
- `--minify`：更接近正式发布后的 bundle
- `-c`：清理缓存，避免旧缓存干扰

这一步通过后，再考虑发布热更新。

---

## 三、每次发布前必须检查

### 1. 页面是否正常打开

- 目标页面能正常进入
- 不白屏
- 不闪退
- 不出现明显布局异常

### 2. 接口是否正常

- 请求能正常发出
- 返回数据符合预期
- 错误提示符合预期
- 成功提示符合预期

### 3. 操作链路是否完整

- 从入口进入路径正常
- 提交操作正常
- 返回上一级正常
- 再次进入页面正常
- 刷新后状态正常

### 4. 数据回显是否正常

- 新提交内容能看到
- 计数变化正确
- 列表刷新正确
- 弹窗关闭后页面状态正确

---

## 四、评论/回复类问题专项检查

如果本次改动涉及：

- 评论
- 回复
- 点赞
- 收藏
- 接口参数组装
- 弹窗状态流转

发布前至少检查以下项目：

- 顶级评论正常
- 二级回复正常
- 回复指定用户正常
- 成功提示正常
- 失败提示正常
- 列表刷新正常
- 评论数量变化正常
- 返回后状态正常
- 再次操作正常

---

## 五、什么时候可以只靠模拟器

以下情况，通常可以先在模拟器验证，再用生产近似模式确认，最后再热更新：

- 纯 JS 页面逻辑
- 接口参数组装
- 列表渲染
- 状态切换
- 文案提示
- 评论、回复、点赞、收藏等业务逻辑

这类问题通常不需要每改一次就上真机热更新。

---

## 六、什么时候不能只靠热更新

如果改动涉及以下内容，不要只发热更新：

- `app.json`
- `android/`
- `ios/`
- Expo plugins
- 原生依赖
- 权限配置
- 原生模块接入

这类改动需要重新构建安装包验证。

---

## 七、推荐工作流

每次改动后，按这个顺序执行：

1. 用开发模式调试
2. 在模拟器跑通主要功能
3. 切到生产服务器复测关键路径
4. 用生产近似模式再测一次
5. 确认没有原生改动
6. 最后再发热更新

---

## 八、常用命令

开发调试：

```powershell
npm run start:dev
```

开发调试（tunnel）：

```powershell
npm run start:dev:tunnel
```

发布前验证：

```powershell
npx expo start --dev-client --no-dev --minify -c
```

---

## 九、简单判断口诀

- 先在模拟器改
- 再用生产近似测
- 通过再发热更新
- 原生改动不走热更新
