# 代码修改规则 - 必须严格遵守

## 核心原则：保护已有修复，避免重复劳动

### 修改代码前的必要步骤（按顺序执行）：

1. **全局搜索相关代码**
   - 使用 grepSearch 搜索相关的函数名、变量名、关键字
   - 检查是否有其他文件已经处理了类似的逻辑
   - 确认问题是否已经在其他地方被修复过

2. **读取完整上下文**
   - 使用 readCode 或 readFile 读取相关文件的完整内容
   - 理解整个模块的逻辑流程，不只看单个函数
   - 确认修改不会破坏已有的修复

3. **最小化修改原则**
   - 只修改必要的代码，不添加冗余逻辑
   - 如果现有代码已经能处理，就不要重复实现
   - 优先使用已有的工具函数和组件

4. **检查依赖关系**
   - 确认修改不会影响其他模块
   - 检查是否有其他文件依赖当前修改的代码
   - 避免破坏已经修复好的功能

### 严禁的行为：

❌ **不要**在没有全面了解的情况下直接修改代码
❌ **不要**重复修复已经解决的问题
❌ **不要**覆盖用户已经修复好的代码
❌ **不要**添加与现有逻辑重复的代码
❌ **不要**只关注单个文件而忽略整体架构

### 正确的工作流程：

1. 用户提出问题
2. ✅ 使用 grepSearch 全局搜索相关代码
3. ✅ 使用 readCode 读取相关文件的完整内容
4. ✅ 分析问题是否已被修复，或是否有现成的解决方案
5. ✅ 制定最小化的修改方案
6. ✅ 确认不会破坏已有修复
7. ✅ 执行修改
8. ✅ 使用 getDiagnostics 验证修改

### 特别注意：

- 如果用户说"不要将我已经修复好的问题又重现出来"，必须先全面检查相关代码
- 如果发现问题可能已经被修复，先询问用户确认
- 修改前要理解整个功能模块的逻辑，而不是只看单个函数
- 保持代码的一致性和简洁性

## 这些规则会在每次会话中自动加载，确保始终遵守
