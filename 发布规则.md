# 发布规则

## 当前策略

- `expo-updates` 已启用。
- `runtimeVersion` 使用显式字符串，默认跟随应用 `version`。
- 当前兼容性规则：
- 相同 `version` -> 属于同一组 OTA 热更新兼容范围
- 不同 `version` -> 属于不同 OTA 热更新兼容范围

相关配置文件：

- [app.json](/E:/qa-native-app/app.json)
- [app.config.js](/E:/qa-native-app/app.config.js)
- [eas.json](/E:/qa-native-app/eas.json)

## 快速结论

1. 只改 JS，不需要发新安装包
- 只发 OTA
- 不修改 `version`
- 不修改 `buildNumber` / `versionCode`

2. 需要发新安装包，但仍希望沿用同一组 OTA 兼容范围
- 构建新安装包
- 保持 `version` 不变
- 只增加构建号

3. 有原生变更，或者需要切换到新的 OTA 兼容范围
- 增加 `version`
- 增加构建号
- 构建新安装包

## 如何判断

按这条规则执行即可：

- 只改了 JS、样式、文案、接口调用、页面逻辑：
- 通常只发 OTA
- 需要重新发商店包、TestFlight 包、内测 APK：
- 只升构建号
- 改了 Expo SDK、原生依赖、`android/`、权限、原生配置：
- 升 `version`，同时升构建号

## 平台规则

### Android

- `versionCode` 用于安装包迭代。
- OTA 兼容性不由 `versionCode` 控制。
- 当前 `production-apk` 已开启 EAS `autoIncrement`，生产构建时会自动增加 Android 构建号。

### iOS

- `buildNumber` 用于安装包迭代。
- OTA 兼容性不由 `buildNumber` 控制。
- 只要要发新的 iOS 安装包，就应该增加 `buildNumber`。

## OTA 使用规则

只有当现有安装包可以安全运行新的 JS Bundle 时，才适合只发 OTA。

适合只发 OTA 的场景：

- 页面样式调整
- 文案修改
- JS 层 Bug 修复
- 接口参数或响应处理更新

不应只依赖 OTA 的场景：

- 升级 Expo SDK
- 新增原生依赖
- 修改原生权限
- 修改 `android/`
- 任何二进制层面的行为变化

上述情况应发布新的安装包，并提升 `version`。

## 版本示例

### 场景 1：只修复页面 JS Bug

- `version`：不变
- `buildNumber` / `versionCode`：不变
- 操作：直接发 OTA

### 场景 2：重新打 Android APK 给测试，但没有原生改动

- `version`：不变
- `versionCode`：增加
- 操作：重新构建 APK

### 场景 3：升级 Expo SDK 或修改原生配置

- `version`：增加
- `buildNumber` / `versionCode`：增加
- 操作：发布新安装包，后续 OTA 只服务于这组新版本

## 推荐团队流程

1. 先判断这次改动是否包含原生变更。
2. 如果没有原生变更：
- 判断是否只发 OTA 就够了。
- 如果只发 OTA，版本号和构建号都不用动。
- 如果需要重新发安装包，只增加构建号。
3. 如果包含原生变更：
- 增加 `version`
- 增加构建号
- 构建并分发新的安装包

## 项目说明

- 当前 `runtimeVersion` 默认跟随 [app.json](/E:/qa-native-app/app.json) 的 `version`。
- 对当前 bare workflow 项目来说，这比 fingerprint 方案更稳定。
- 如需手动覆盖 `runtimeVersion`，仍可通过环境变量 `EXPO_RUNTIME_VERSION` 指定，但默认发布流程通常不需要这样做。
